# UX state sync, toast notifications, and stealth naming

This change adds realtime IPC-driven state synchronization between the main process and settings views (HotkeysView, CustomizeView), a toast notification overlay that confirms hotkey actions, Ctrl+Enter triggering the start button's visual animation, font weight as a new CSS variable, a lower opacity floor (0%), and stealth process renaming. The bulk of the diff is CSS reformatting from single-line to multi-line rules in ApiKeysView — behaviorally irrelevant but noisy.

**Watch for:**

- Ambiguous toast messages — both window opacity and background opacity display `"Opacity: X%"` (confirmed)
- Window fully invisible at 0% opacity with no guaranteed recovery path if user forgets the hotkey (confirmed)
- IPC listener leak in CustomizeView — `bg-opacity-updated` event listener is registered on the app element but never removed on disconnect (confirmed)

## High-level view

The state sync approach is straightforward: IPC events emitted by the main process (`scale-changed`, `zoom-changed`, `opacity-changed`, `font-size-changed`, `voice-toggled`) are consumed in the renderer by both the global toast system and by settings views that mirror the values in sliders. HotkeysView registers listeners in `connectedCallback` and cleans them up in `disconnectedCallback`. CustomizeView does the same for font-size but uses a mixed strategy for background opacity — listening on the app element via a custom DOM event rather than IPC — and neglects to remove that listener.

The toast system lives in renderer.js as a module-scoped singleton. It creates a fixed-position container on first use, replaces its content on each call (so only one toast is visible at a time), and auto-hides after 1.5 seconds with a fade-out animation. The system reads a persisted preference `hotkeyToastsEnabled` and can be toggled from CustomizeView via `setHotkeyToastsEnabled`.

Ctrl+Enter reaches into shadow DOM to `.click()` the start button, triggering the aurora/dither canvas animation rather than calling `handleStart()` directly. The fallback chain is `mainView.shadowRoot.querySelector('.start-button')` then `appEl.handleStart()`.

The stealth naming renames the Electron app from "Cheating Daddy" to "Service Host" / "Service Host: Network Service" in package.json and all forge maker configs, making the process and window title blend in with system services.

## Ambiguous opacity toast messages

Two distinct IPC handlers produce identical toast text. The `bg-opacity-change` handler in renderer.js (line ~1247) displays `Opacity: ${Math.round(next * 100)}%` after adjusting the CSS background transparency. Separately, the `opacity-changed` listener (line ~1283) also shows `Opacity: ${Math.round(val * 100)}%` after the main process changes `BrowserWindow.setOpacity`. A user pressing Ctrl+Shift+[ (background) or the opacity hotkeys (window) will see the same toast label despite controlling different things.

```js
// bg-opacity-change handler
showToast(`Opacity: ${Math.round(next * 100)}%`);

// opacity-changed handler
showToast(`Opacity: ${Math.round(val * 100)}%`);
```

These should use distinct labels, e.g. `"Background: X%"` and `"Window Opacity: X%"`.

## Event listener leak in CustomizeView

In the constructor, CustomizeView attaches `_bgOpacityListener` directly on the app DOM element:

```js
const app = document.querySelector('cheating-daddy-app');
if (app) app.addEventListener('bg-opacity-updated', this._bgOpacityListener);
```

The `disconnectedCallback` cleans up IPC listeners stored in `_ipcCleanups` but never removes this DOM event listener. If the view is mounted/unmounted multiple times (e.g., navigation), listeners accumulate. The fix is to store a reference to the app element and call `removeEventListener('bg-opacity-updated', this._bgOpacityListener)` in `disconnectedCallback`.

## Zero-percent window opacity with no safety net

`HARD_LIMITS.opacityMin` drops from `0.2` to `0.0` in window.js, and `DEFAULT_WINDOW_STATE.opacityMin` is also `0.0`. The opacity hotkey handler in the main process clamps to this floor, meaning a user can make the window fully invisible. Since globalShortcuts are registered on the OS (not the window), the user can theoretically recover by pressing opacity-up. However, if the user is unfamiliar with the shortcut or if `globalShortcut.register` fails silently (which Electron permits), there is no fallback — the app becomes permanently invisible until the process is killed. A floor of 0.02–0.05 would prevent an unrecoverable state while still being visually nearly invisible.
